The ITAC Operations & Client Portal

Overview

This project involves designing and building a unified web portal for OSU's Industrial Training and Assessment Center (ITAC) to manage the entire client assessment lifecycle. The existing process relies on disconnected spreadsheets, emails, and manual data entry, leading to inefficiencies, errors, and a poor experience for both internal staff and external clients.

My Role: UX Engineer (Lead Designer & Full-Stack Developer)
Timeline: Oct 2025 - Present (Ongoing)
Tools: Figma, Django, Python, JavaScript, PostgreSQL

The Problem

ITAC's assessment process is complex, involving multiple stakeholders (staff, management, student assessors, clients). The current workflow is manual and fragmented, causing several key problems:

  • **Staff Inefficiency:** Hours are wasted on duplicate data entry and tracking project status via email.
  • **Client Frustration:** Clients have no central place to view their project status, submit documents, or see final reports.
  • **Data Errors:** Manual data transfer between spreadsheets leads to frequent errors and version control nightmares.
  • **Lack of Oversight:** Management has no simple dashboard to view project load, team capacity, or financial status.

The Process & Solution

As the sole UX Engineer, I am responsible for the entire product lifecycle. The solution is a secure, role-based web portal that provides a single source of truth for all stakeholders.

User Research & Personas

I conducted stakeholder interviews with 4 key user groups: ITAC Staff, Management, Student Assessors, and External Clients. This led to the creation of 4 primary personas, each with distinct needs and pain points. For example, the "Staff" persona's main goal is "time-saving and efficiency," while the "Client" persona's main goal is "clarity and transparency."

User Flow & Wireframes

I mapped out the end-to-end user flow for the assessment process, from a client's first inquiry to the final report delivery. This visual map became the blueprint for the portal's architecture. I then created low-fidelity wireframes in Figma to block out the key screens for each user role, focusing on the "Staff Dashboard" and "Client Project View" first.

High-Fidelity Design & Prototyping

Using the main OSU brand guidelines, I created a clean, accessible, and professional high-fidelity design system in Figma. I built an interactive prototype focusing on the primary task: "Creating a new client project and assigning a team." This prototype was used for initial usability testing with 3 staff members.

Development & Implementation

This is where my "Dev-Designer" role is critical. I am currently building the portal from the ground up as a full-stack application.

  • **Backend:** Django (Python) for the server, handling user authentication, permissions, and the database models.
  • **Database:** PostgreSQL for robust data management.
  • **Frontend:** Vanilla JavaScript, HTML, and CSS (using the `index.css` theme as a base) to build the interactive user interface based on my Figma designs.

What I've Learned (So Far)

This ongoing project is a massive lesson in balancing the needs of multiple, diverse user groups. The "Client" needs extreme simplicity, while the "Staff" needs powerful, complex tools. The biggest challenge is designing a single system that serves both without compromise. Building the front-end directly from my own designs has also been incredibly efficient, allowing me to spot and fix usability issues during development that I might have missed in a static prototype.
